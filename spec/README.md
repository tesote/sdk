# spec/

Frozen, vendored API contract used by every language under `packages/`. `v1.openapi.yaml` and `v2.openapi.yaml` are byte-for-byte copies of the upstream `openapi.yaml` / `openapi_v2.yaml` shipped from the Rails engine `engines/tesote_api/docs/` in the `tesote/treasury` repo, prefixed with a five-line provenance header. `v3.openapi.yaml` is a derived stub (`info.x-status: derived-stub`) extracted from the v3 routes block and `app/controllers/tesote_api/v3/*` since upstream has no v3 OpenAPI doc yet — schemas are loose where the jbuilder views weren't statically readable, flagged inline with `# stub:` comments. `parity.yaml` is the hand-curated cross-language manifest (resources + typed errors), `error_codes.txt` is the sorted list of `error_code` envelope values, and both are consumed by `bin/check_parity.py` / `.github/workflows/parity-check.yml` to assert each SDK exposes everything the manifest names.

To refresh: `cd ../treasury && git rev-parse HEAD` for the new SHA, then re-run the vendoring (`cp engines/tesote_api/docs/openapi*.yaml ../sdk/spec/` and re-add the header — see the existing files for the exact five-line format). For `v3`, re-read every controller under `engines/tesote_api/app/controllers/tesote_api/v3/` and adjust `v3.openapi.yaml` by hand; do not invent fields. Re-extract `error_codes.txt` from `engines/tesote_api/app/lib/tesote_api/error_codes.rb` (`grep -oE "'[A-Z_]+'" ... | tr -d "'" | sort -u`). Whenever resources or errors change, update `docs/architecture/resources.md` / `docs/architecture/errors.md` and `parity.yaml` in the same PR — `parity-check.yml` will fail until every SDK catches up.
